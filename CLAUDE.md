# Touch Aim - Claude Code handoff

このファイルを最初に読み、次に `index.html` を確認すること。

## 現在のプロジェクト

- スマホ中心のタッチエイムゲーム。
- 実装は現状 `index.html` 1ファイル（HTML/CSS/JavaScript）。
- 25秒以内に10個の円をタップし、反応時間・中心精度・円サイズ・コンボから得点を計算する。
- 3モードは出現ルールではなく配点だけが違う。表示名は全言語共通の英語。
  - `normal` = `BALANCED`: 速さと精度のバランス
  - `acc` = `PRECISION`: 精度重視
  - `speed` = `SPEED`: 速度重視
- 記録はバージョン付きストレージ `touchaim_v2`（schema:2）に一元化済み。旧 `touchaim_*` キーからの移行実装済み（旧キーは安全のため残置）。累計 plays/hits も蓄積開始済み（実績の下地）。
- 単体／通しリプレイ、散布図、10言語対応あり。
- 2026-07-18時点: 4分類メニュー実装済み。SCORE ATTACK は移設完了、TRAINING / ASSESSMENT / PROFILE は「準備中」スタブ。
- タイトル上部の余白調整（h1 margin-top: 言語セレクタとの重なり回避）はコミット済み。

## 今回決まった大分類

画面と機能を次の4分類に再構成する。

1. `SCORE ATTACK`
   - 現在のゲームをここへ移す。
   - 子モード名は `BALANCED` / `PRECISION` / `SPEED`。
   - 実績の8,000、9,000、10,000点はここで判定する。
2. `TRAINING`
   - 下記6能力を個別に練習できる。
3. `ASSESSMENT`
   - 6種目を通して能力を測定する5～7分程度のテスト。
   - 1日2回まで。日付変更時にリセット。
   - 2回分を両方保存し、その日の代表値は2回の平均（1回だけならその値）。
   - 分析用データでは原則その日の1回目を使い、練習効果による偏りを抑える。
   - 中断時は一定時間内なら続きから再開できる設計が望ましい。
4. `PROFILE`
   - 六角形レーダーチャート、能力別の説明、過去との比較を表示する。

`QUICK TRAINING` は採用しない。勝手に追加しないこと。

## エイム能力の6パラメーター

名称と意味は以下で固定する。各軸は別課題で測り、単なる同一スコア式の重み替えにしない。

### 1. PERCEPTION

- 標的を視認して動作を開始する速さ。
- 中央の待機エリアに指を置き、ランダム時間後に大きな標的を表示。
- 出現から指が待機エリアを離れるまでを反応時間として測る。
- 見落とし、中央値、ばらつきを記録。

### 2. FLICK

- 離れた標的へ一気に指／照準を運ぶ能力。
- 待機位置から距離・方向・大きさを均等化した静止標的へ移動。
- 反応開始後の移動時間、距離、サイズ、最初の着弾誤差を測る。
- 距離とサイズの難易度差を無視して単純な時間だけで採点しない。

### 3. PRECISION

- 狙った位置へ正確に止める能力。
- 小型ターゲットを固定間隔で表示し、速度の比重を下げて中心精度を測る。
- 平均中心誤差、誤差分散、縦横の偏り、ミス率を記録。

### 4. TRACKING

- 動く対象を継続して追う能力。
- 指を画面につけたまま、移動ターゲットを一定時間追従。
- ターゲット内滞在率、中心からの平均距離、外れてからの復帰時間、軌跡の滑らかさを測る。

### 5. SWITCHING

- 複数対象を素早く正しく切り替える能力。
- 複数ターゲットを同時表示し、指定された順番にタップ。
- 切替時間、長短距離差、誤タップ、連続成功数を測る。

### 6. TIMING

- 動体の未来位置を読んで当てる能力。
- 移動ターゲットを追い続けず、一度のタップで迎撃。
- 位置誤差、早押し／遅押し、速度変化・方向転換への対応を測る。

重要: 現在の直接タップ方式は「視覚ターゲット捕捉＋指の到達能力」であり、FPSのスワイプ照準そのものではない。将来、本格的なスマホFPS向けを名乗る場合は、ドラッグでクロスヘアを動かす入力方式やジャイロを別途検討すること。現段階の名称は `Touch Aim Profile` とする。

## ASSESSMENTの採点とデータ設計

- 当初は暫定的な0～100点換算を実装する。
- 最速1回ではなく、複数試行の中央値とミス率を使う。
- 方向、距離、サイズ、速度の構成を毎回均等にし、順番だけをランダム化する。
- ランダムな有利不利を避けるため、評価用セットはseedまたは固定構成バージョンを持つ。
- 六角形の合計点だけではなく、生データを保存する。
- 必須バージョン:
  - `assessment_version`
  - `test_config_version`
  - `score_formula_version`
- 端末差確認用に、個人を特定しない範囲で画面サイズ、向き、DPR、リフレッシュレート、OS／ブラウザ大分類を保持できる構造にする。
- β前はローカルJSON/CSVエクスポートで知人テストを可能にする。
- 知らないテスターへ配る段階で、明示同意を得た匿名送信を追加する。氏名、メール、正確な位置情報は収集しない。
- 後から換算式を変更し、過去の生データから全スコアを再計算できること。
- 暫定→知人等20～50人程度でβ基準調整→100～300完走データで分布作成→500～1,000以上で安定化、という段階を想定（人数は目安）。

## ローカル実績（25個）

すべて端末内保存。解除時に短い演出を出し、一覧では解除数 `ACHIEVEMENTS x/25`、解除日時、未解除状態を表示する。設計目標は「上級者でも初回に大量解除されないこと」。単純な平均値や1回のノーミスだけを大量に並べず、全10発、複数回、全モード、累計で条件を作る。同一プレイで複数条件を満たした場合も解除は可能だが、同系統の段階実績は1プレイにつき上位1個だけ通知し、解除演出が連続しすぎないようにする。

1. `SYSTEM ONLINE` — 初めて1プレイ完了
2. `CLEAN RUN` — 空振りなしのラウンドを累計5回完了
3. `PERFECT CLEAR` — 10/10ヒットを3ラウンド連続で達成
4. `FLAWLESS EXECUTION` — 空振り・タイムアウトなしの完全クリアを3モードすべてで達成
5. `SHARPSHOOTER` — 1ラウンドの全10発をそれぞれ精度80%以上で命中
6. `DEAD CENTER` — 1ラウンドの全10発をそれぞれ精度90%以上で命中
7. `ZERO DEVIATION` — 平均精度95%以上、かつ全10発が精度90%以上
8. `QUICK HAND` — 10/10ヒットかつ平均反応時間400ms以下
9. `LIGHTNING` — 10/10ヒットかつ平均反応時間300ms以下
10. `COMBO STARTER` — 中心コアへのヒットを累計50回
11. `CORE MASTER` — 10コンボを3ラウンド連続で達成
12. `HIGH SCORER` — 3モードすべてで8,000点以上
13. `BREAKPOINT` — いずれかのモードで9,000点以上
14. `OVERDRIVE` — 3モードすべてで9,000点以上
15. `TRANSCENDENCE` — いずれかのモードで10,000点以上
16. `S RANK` — Sランクを3ラウンド連続で獲得
17. `BALANCED MASTER` — BALANCEDでSランクを累計3回獲得
18. `PRECISION MASTER` — PRECISIONでSランクを累計3回獲得
19. `SPEED MASTER` — SPEEDでSランクを累計3回獲得
20. `TRINITY MASTER` — 上記3つのモードマスター実績をすべて解除
21. `TARGET HUNTER` — 累計100ヒット
22. `TARGET BREAKER` — 累計300ヒット
23. `TARGET REAPER` — 累計500ヒット
24. `TARGET SOVEREIGN` — 累計1,000ヒット

累計ヒット数、実績、測定回数などを保存するため、散在したキーを増やすのではなく、バージョン付きのストレージスキーマと移行処理を設計すること。既存のベスト・履歴・TOP10を失わないこと。

## CrazyGamesと収益化の扱い

### Basic Launchまで

- 全機能を無料で遊べるようにする。
- 広告、★消費、広告チャージによる制限を有効化しない。
- ASSESSMENTの1日2回制限は広告と無関係に有効。
- Basic Launchは広告が無効なので、広告がないと進めないUIを置かない。

### Full Launch後の構想（今は有効化しない）

- 2回目のASSESSMENTを広告または★で開始できる案。
- 毎日★を配り、TRAININGまたはSCORE ATTACKで消費する案。
- ユーザーの初期案は「毎日★5、広告視聴で★5チャージ、1日3回まで」。消費量や無料プレイ数はまだ最終決定ではない。
- ★切れと広告頻度が平均プレイ時間を壊さないか、Full Launch後の実測を見て決める。
- CrazyGames版ではCrazyGames SDK以外の広告を入れない。
- 収益化はfeature flag／ホスト判定で切り替え、広告利用不可時もゲーム全体が正常動作するようにする。

## 現行ソースを読んだ上での改善点

新機能と同時に壊さないよう、以下を優先度順に処理または設計へ反映する。

### 優先度: 高

1. **1ファイル巨大化への対処**
   - これ以上 `index.html` に全機能を直書きすると保守不能になる。
   - 既存挙動を回帰確認しながら、状態管理、ストレージ、i18n、SCORE ATTACK、TRAINING、ASSESSMENT、PROFILE、リプレイを分離する。
   - 配布時に単一HTMLが必要なら、ソースを分割してビルド時にまとめる構成でもよい。
2. **SCORE ATTACKの待ち時間** — 対応済み (2026-07-18)
   - ターゲット解決後の短いランダム間隔（べき分布・合計4.5秒固定）で次を出す方式に変更済み。
   - 高速プレイならラウンドは約7秒で終わる。全タイムアウトの最悪ケースでも25秒に収まる。
3. **ASSESSMENTで反応時間と移動時間を分離**
   - 現在の `spawnAt` から `pointerdown` までは視認・判断・移動が混ざった総時間。
   - PERCEPTION／FLICKでは中央待機点とpointerイベントを使い、動作開始と着弾を分けて記録する。
4. **ストレージの一元化と移行**
   - 現在は `touchaim_*` キーが機能ごとに散在。
   - バージョン付きデータ構造、破損時のフォールバック、エクスポート／インポートを用意し、既存データを移行する。
5. **評価条件の再現性**
   - `Math.random()` だけに依存せず、評価課題にはseedと構成バージョンを導入する。

### 優先度: 中

6. **画面回転・狭い横画面への対応**
   - ターゲット座標計算は高さが不足する横画面で負の範囲になり得る。
   - `resize` / `orientationchange`、safe area、最小プレイ領域を扱う。
   - リプレイは記録時と再生時の画面寸法が変わった場合に座標を正規化して再描画する。
7. **i18nの方針（確定・ユーザー合意済み 2026-07-18）**
   - `BEST SCORE`、`RECORDS`、`MISS`、`TIME OUT`、評価語（EXCELLENT等）、`COMBO`、モード名（BALANCED等）、大分類名（SCORE ATTACK等）は**ゲーム語彙として全言語共通の英語で固定**。未対応ではなく意図的。
   - 説明文・ラベル類はi18n対象（現在10言語、`STR`辞書 + `data-i18n`属性方式）。
   - 新画面追加時もこの方針を維持する。
8. **端末・入力条件の記録**
   - 測定結果は端末差の影響を受ける。画面寸法、向き、DPR、推定Hz、pointerTypeを記録し、異なる条件を無条件に比較しない。
9. **サウンド・振動・モーション設定**
   - 音、振動のON/OFFと `prefers-reduced-motion` 対応を追加する。
10. **外部フォント依存**
   - Google Fontsが失敗しても遊べるが、読み込み・プライバシー・オフライン性に影響する。セルフホストまたは現状のフォールバックを明確にする。
